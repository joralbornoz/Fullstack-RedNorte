Patrón 01 Arrange · Act · Assert // estructura universal de cada test
Cada prueba se divide en tres bloques con responsabilidades separadas. El color de borde izquierdo indica qué fase está activa — sin mezclar preparación, ejecución y verificación.
📄 LoginPage.test.jsx
test('muestra error cuando las credenciales son inválidas', async () => {

Arrange — preparar estado inicial
  // El mock simula que el servidor rechaza las credenciales
  authService.login.mockRejectedValue({
    response: { data: { mensaje: 'RUT o contraseña incorrectos' } },
  })
  renderLogin()    // monta el componente en el DOM de prueba

Act — simular acciones reales del usuario
  await userEvent.type(screen.getByPlaceholderText('Ej: 12.345.678-9'), 'rut-malo')
  await userEvent.type(screen.getByPlaceholderText('••••••••'), 'clave-mala')
  await userEvent.click(screen.getByRole('button', { name: /ingresar/i }))

Assert — verificar el resultado esperado
  await waitFor(() =>
    expect(screen.getByText('RUT o contraseña incorrectos')).toBeInTheDocument()
  )
  expect(mockNavigate).not.toHaveBeenCalled()

})

Patrón 02 Mocking Estratégico — Test Doubles // dos estrategias según el contexto
Estrategia A: jest.mock() de módulo completo para servicios externos (nunca se ejecuta Axios real). Estrategia B: Provider real + componente consumidor para probar la lógica interna de React Context.

📄 AuthContext.test.jsx — Estrategia B: Provider real + spy
// Para testear la lógica interna del Context, usamos el Provider real
// y un componente consumidor que actúa como "espía"

function TestConsumer() {
  const { user, login, logout } = useAuth()
  return (
    <>
      <span>{user?.nombre ?? 'sin sesión'}</span>
      <button onClick={() => login({ nombre: 'Ana', rol: 'ADMIN' })}>Login</button>
    </>
  )
}

test('login() guarda el usuario y lo persiste en localStorage', async () => {
  render(<AuthProvider><TestConsumer /></AuthProvider>)

  await userEvent.click(screen.getByText('Login'))

  expect(screen.getByText('Ana')).toBeInTheDocument()
  expect(localStorage.getItem('rednorte_user')).toContain('Ana')
})
mockResolvedValue — camino exitoso
mockRejectedValue — camino de error
Provider real — para lógica interna de Context
Patrón 03 Factory Method — renderAndLoad() // evita repetir configuración en cada test
Sin el factory, cada test necesitaría 8+ líneas para configurar Router, Context y esperar carga. La función centraliza eso: cada test usa 1 línea y se enfoca solo en su lógica.
📄 AdminDashboard.test.jsx
// ── La función Factory ────────────────────────────────────────────────
const USER_MOCK = { nombre: 'Admin Test', rol: 'ADMIN', token: 'jwt-ok' }

async function renderAndLoad(overrides = {}) {
  useAuth.mockReturnValue({ user: { ...USER_MOCK, ...overrides }, logout: jest.fn() })

  render(
    <MemoryRouter initialEntries={['/admin']}>
      <Routes>
        <Route path="/admin" element={<AdminDashboard />} />
      </Routes>
    </MemoryRouter>
  )

  // Espera a que el spinner de carga desaparezca antes de continuar
  await waitFor(() =>
    expect(screen.queryByText(/cargando/i)).not.toBeInTheDocument()
  )
}

// ── Uso: cada test queda en 2-3 líneas ───────────────────────────────

test('muestra la lista de pacientes al cargar', async () => {
  await renderAndLoad()
  expect(screen.getByText('Ana Pérez')).toBeInTheDocument()
})

test('muestra error cuando la API falla', async () => {
  listaEsperaService.getAll.mockRejectedValue(new Error())
  await renderAndLoad()
  expect(screen.getByText(/error de conexión/i)).toBeInTheDocument()
})

test('modal valida campos vacíos al enviar', async () => {
  await renderAndLoad()
  await userEvent.click(screen.getByText('+ Nueva consulta'))
  expect(screen.getByText('El RUT es requerido')).toBeInTheDocument()
})
renderAndLoad() — factory que construye todo el entorno
overrides — permite variaciones por test sin nueva función
